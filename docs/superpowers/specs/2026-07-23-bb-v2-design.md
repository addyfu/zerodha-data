# bb_mean_reversion_v2 — Design Spec (Pre-Registered)

Date: 2026-07-23. Status: APPROVED (user: add-only, never replace the original).
Origin: live process review of 2026-07-23's trades found bb_mean_reversion
repeatedly placing noise-width stops (three trades with stops ≤0.3% of entry,
one at ~0.0%) and re-entering a just-stopped symbol within the hour with a
tighter stop. Those observations are HYPOTHESES about design flaws — this spec
tests them; it does not assume them.

## The variant (frozen — exactly two changes from the original)

bb_mean_reversion_v2 is a copy of bb_mean_reversion with:

1. **Stop floor:** stop_loss = whichever is WIDER of (a) the original band-based
   stop, (b) 0.5% from entry. Take-profit logic unchanged (note: this can
   reduce the reward:risk ratio below the detector's 1.5 gate for setups whose
   target was predicated on a hair-width stop — such signals will now be
   rejected by the existing gate; that rejection is part of the variant's
   behavior, not a bug).
2. **Post-stop cool-down:** after a stop-loss/trailing-stop exit in a symbol,
   no new v2 entry in that same symbol for 60 minutes.

Nothing else may differ. Entries, bands, sizing inputs, EOD behavior: identical.

## Head-to-head screen (before any incubator seat)

- Harness: the intraday probe conventions (kite/research/intraday_probe.py) —
  1-min data, next-bar entries, intraday Zerodha costs + 0.05% slippage/side,
  15:15 square-off, max 5 concurrent, one trade per symbol per day per variant.
- Samples: (1) Nov 2025 – Jan 2026 43-day minute CSVs, (2) Jul 13-20 2026
  release-DB week. Both variants run on identical data.
- Signal source: the actual BBMeanReversion strategy class from
  kite/strategies/ generating signals on 5-min resampled bars (the strategy's
  live timeframe), executed on 1-min bars.

**Frozen verdict — v2 earns an incubator seat only if ALL hold:**
1. v2 total net P&L > original's on BOTH samples
2. v2 max single-trade loss no worse than original's on both samples
3. v2 trade count ≥ 40% of original's (the floor+cooldown must not
   effectively kill the strategy — a variant that barely trades proves nothing)

Any failure → v2 is discarded, the observation is recorded as "tight stops
appear load-bearing (or floor too blunt)", and the original's process grades
get a written correction. No threshold changes after results are seen.

## Caveats (stated before results)

- The minute-data evidence base is thin for BOTH variants — this screen
  filters obviously-worse, it does not prove better. The live race decides.
- Two changes are bundled; if v2 wins we won't know which change mattered.
  Accepted: the bundle IS the hypothesis ("sane exits"), and unbundling would
  double the test surface on data too thin to support it.
