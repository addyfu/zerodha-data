# Parity Monitor — Design Spec (Pre-Registered)

Date: 2026-07-21. Status: DRAFT — awaiting user approval before build.
Companion to the self-auditing system discussion; this is its first and most
valuable component. Discipline: thresholds frozen here, before any check runs;
amendments require a dated decision-log entry.

---

## 1. Problem statement

Every failure in this project's history shared one shape: **the machine broke
silently while looking normal.**

| incident | silent for | would parity catch it? |
|---|---|---|
| Signal enum bug — 444 runs, zero trades | ~3 months | day 1-3 (trade-rate check) |
| Placeholder enctoken — all fetches HTTP 400 | 5+ weeks | day 1 (liveness + data checks) |
| fib_3wave structurally unable to fire | forever | week 1 (trade-rate check) |
| TOTP re-login loop every 5 min | hours | day 1 (auth-event count) |
| Entries after square-off holding overnight | 1 day | day 1 (position-at-close check) |
| Missed July rebalance (empty daily cache) | would've been 1 month | day 1 (rebalance-event check) |

The monitor's sole job: make silence impossible. Every strategy carries a
frozen "Expectation Book" from its backtest; every evening a pure-arithmetic
script compares the day's reality against it and reports GREEN/AMBER/RED.
No AI anywhere in the loop — a smoke detector must be deterministic.

## 2. Non-goals

- Never places, modifies, or exits trades. Only side-effects: Telegram
  messages and writing `strategies_paused.json` (which the live monitor
  already consults before ENTRIES only).
- Not a performance judge. Its thresholds detect *implementation breakage*
  (behavior ≠ backtest), not *edge decay* (that's the monthly scorecard's
  job, out of scope for v1).
- No auto-unpause. A human clears every RED.

## 3. Components

### 3.1 Expectation Book (`kite/live_monitor/expectations/<strategy>.json`)

One frozen JSON per live strategy, generated by `gen_expectations.py` from the
strategy's own validation backtest (honest_lab for momo_rotation_63,
retest_all for the swing candidates, walkforward sample for the intraday
incubator set — wide bands where the backtest was weak, because these cards
detect bugs, not judge alpha):

```json
{
  "strategy": "momo_rotation_63",
  "book": "main",
  "cadence": "monthly",
  "trades_per_month": {"lambda": 4.6, "hard_min_window_days": 45},
  "win_rate": {"p": 0.42, "n_backtest": 199},
  "avg_win_pct": 7.1, "avg_loss_pct": -3.4,
  "max_drawdown_pct": -29.7,
  "hold_days": {"p50": 31, "p95": 92},
  "slippage_assumed_pct": 0.05,
  "generated": "2026-07-21", "source": "honest_lab ext 2015-2026"
}
```

Regenerating a card requires a git commit (cards are tracked files) — no
silent expectation drift.

### 3.2 Nightly check run (`kite/live_monitor/parity_monitor.py`)

Runs on Oracle via systemd timer at **16:15 IST Mon-Fri** (after EOD close and
daily summary). Reads: both paper-trade DBs, expectation JSONs, monitor.log
tail, `data/momo_rotation_state.json`, announcement-filter status. Writes:
one Telegram line (always), `parity_history.jsonl` (always), and
`strategies_paused.json` (on RED with pause action).

### 3.3 Checks and frozen thresholds

| # | check | applies to | AMBER | RED | RED action |
|---|---|---|---|---|---|
| P1 | **Liveness**: scan cycles logged today (market day) | system | — | zero cycles | alert (nothing to pause — monitor is dead) |
| P2 | **Data freshness**: daily cache count at swing scan | system | <90% of universe | <50% | alert |
| P3 | **Trade rate**: Poisson two-tail prob of observed trade count vs λ over rolling 20 trading days (intraday/swing strategies only) | per strategy | p<0.05 | p<0.01 | pause entries |
| P4 | **Cadence events**: momo rebalance occurred within first 3 trading days of month; EOD square-off left zero INTRADAY positions open past 15:30 | momo / books | — | violated | alert (+pause for repeat) |
| P5 | **Win rate**: once live N≥60 closed trades — two-proportion z-test live vs card | per strategy | p<0.05 | p<0.01 | pause entries |
| P6 | **Drawdown**: live equity DD vs card max_drawdown | per book | >1.5× | >2.0× | pause entries |
| P7 | **Hold time**: live p50 hold vs card p95 | per strategy | p50 > card p95 | — | amber only |
| P8 | **Slippage/fill sanity**: paper fill price vs same-day candle range (fill outside [low,high] = impossible fill) | per trade | 1 event | ≥2 events/week | pause entries (harness bug) |
| P9 | **Auth events**: "Login successful" count in today's log | system | >3 | >10 | alert (login loop) |
| P10 | **Ann-filter**: refresh failed | system | 1 day | ≥3 consecutive days | alert |
| P11 | **Ann-filter category drift**: 0 red-flag matches across universe for 15 consecutive market days (historical base rate ≈ daily) | system | — | triggered | alert (category likely renamed) |
| P12 | **Parity dead-man**: parity_history.jsonl last entry age, checked by the NEXT run; plus the daily line itself is the human-visible heartbeat | self | — | gap > 1 market day | noted in next report |

N-gates: P3 activates after 10 trading days of a strategy's live history; P5
after 60 closed trades; P6 after 20 trading days. Before gates: reported as
"warming up", never colored.

### 3.4 Output contract

- **Daily, always, exactly one Telegram line** (the dead-man heartbeat):
  `🩺 parity: 5 strategies green` or
  `🩺 parity: RED momo_rotation_63 (trade-rate p=0.003) — entries paused; 4 green`.
- AMBERs: included in the daily line only as a count; details in Saturday
  digest (with the week's history) to avoid alert fatigue.
- REDs: immediate detail line + action taken + what a human should check.
- Everything appended to `parity_history.jsonl` for the future monthly
  scorecard to consume.

### 3.5 Pause mechanics

`strategies_paused.json`: `{"momo_rotation_63": {"reason": "...", "since": "..."}}`.
Live monitor consults it before entries (30-line hook, same pattern as the
announcement filter). Exits and stop-losses are NEVER paused. Unpause =
human deletes the entry (or tells Claude to). Every pause/unpause appended to
parity_history.jsonl.

## 4. Build plan

| step | what | effort |
|---|---|---|
| 1 | `gen_expectations.py` + 7 cards (momo, 2 swing candidates, 4 incubator) committed | ~1 hr |
| 2 | `parity_monitor.py` checks P1-P12 + Telegram + history file | one evening |
| 3 | monitor.py pause-file hook on all entry paths | 30 min |
| 4 | systemd service+timer on Oracle, 16:15 IST; dry-run week in log-only mode (no pause writes) before arming pause action | 30 min + 1 week soak |
| 5 | Saturday digest (append to existing weekly rhythm) | 30 min |

Total: ~1 evening of build + 1 week of log-only soak before REDs can pause.
The soak protects against false-positive teething (thresholds act only after
observed clean on real days).

## 5. Premortem (what could go wrong with the monitor itself)

1. **False REDs from small samples** → N-gates above; soak week; pause is
   entries-only and reversible in one message.
2. **Alert fatigue → ignored reds** → hard cap: one daily line; ambers
   batched weekly; only REDs get detail lines.
3. **Card drift / stale expectations** (strategy edited, card not regenerated)
   → gen_expectations stamps the source-code git hash of the strategy into
   the card; parity flags hash mismatch as AMBER "card stale".
4. **Parity itself dies silently** → the daily line IS the heartbeat; its
   absence is user-visible, and P12 self-checks on the next run.
5. **Timer timezone drift** → Oracle already runs IST (verified); timer uses
   system tz.
6. **DB schema changes break parsing** → checks fail loud (RED "parity
   internal error"), never fail silent — top-level try/except that itself
   sends the failure line.

## 6. Amendment rule

Thresholds in §3.3 are frozen. Changes require a dated decision-log entry in
this file stating what changed, why, and what data prompted it — same
protocol as the news-project spec. Post-incident loosening of a threshold
that fired correctly is forbidden.
